# Build Guide — SAP Service Layer OpenAPI Spec

This document explains how the files in `assets/spec/` were created and how to
regenerate them.

---

## Source

The spec was parsed from the SAP Help Portal HTML page:

```
https://help.sap.com/doc/056f69366b5345a386bb8149f1700c19/10.0/en-US/Service%20Layer%20API%20Reference.html
```

That page is a static HTML document (not a live Swagger UI loading an external
JSON file).  Every operation is pre-rendered inside a tree of
`<li class="resource">` → `<div class="method">` → `<li class="X operation">`
elements.

---

## Output layout

```
assets/spec/
├── openapi.yaml          # root spec; lists every path with a $ref
└── paths/
    ├── Orders.yaml                         # GET /Orders, POST /Orders
    ├── Orders_id.yaml                      # GET/PATCH/DELETE /Orders({id})
    ├── Orders_id_Close.yaml                # POST /Orders(id)/Close
    ├── OrdersService_GetApprovalTemplates.yaml
    └── ...                                 # ~1 147 files total
```

Each path file contains only the HTTP-verb operation objects (no `paths:` wrapper),
matching the OpenAPI 3 `$ref` format used by `openapi.yaml`.

## Regenerating the spec

### Live fetch (default)

```bash
cd /path/to/skills/skills/sap-service-layer-expert
python scripts/build_spec.py
```

The script fetches the SAP Help Portal page, parses all 499 resource blocks,
and writes the YAML files.  Typical run time: 20–40 s (mostly network).

### Save the HTML for offline use

```bash
python scripts/build_spec.py --save-html /tmp/sap_sl_api_ref.html
```

### Rebuild from a saved HTML file (no network)

```bash
python scripts/build_spec.py --input /tmp/sap_sl_api_ref.html
```

### Regenerate ENDPOINTS.md after a spec rebuild

```bash
python scripts/build_endpoints_md.py
```

## How the HTML is parsed

### Page structure

```
<li class="resource">           ← one per named resource/service (499 total)
  <div class="heading">
    <h2><a>Orders</a></h2>      ← resource name
  </div>
  <div class="methods">
    <p>Description …</p>        ← resource-level description (→ x-resource-description)
    <div class="method">        ← one per HTTP verb × path combination
      <ul class="endpoints">
        <li class="endpoint">
          <ul class="operations">
            <li class="get operation">          ← or post/patch/delete
              <div class="heading">
                <span class="http_method"><a>GET</a></span>
                <span class="path"><a>Orders</a></span>   ← URL path fragment
              </div>
              <div class="content">
                <p>Retrieve a collection …</p>  ← operation description
                <h4>Example</h4>
                <pre>
GET https://localhost:50000/b1s/v2/Orders     ← x-example-url (first line)

{                                              ← request body JSON (if present)
    "CardCode": "c001",
    …
}
                </pre>
              </div>
            </li>
          </ul>
        </li>
      </ul>
    </div>
  </div>
</li>
```

### Path-to-filename mapping

| HTML path fragment        | YAML filename                    | `openapi.yaml` key              |
|---------------------------|----------------------------------|---------------------------------|
| `Orders`                  | `Orders.yaml`                    | `/Orders`                       |
| `Orders(id)`              | `Orders_id.yaml`                 | `/Orders({id})`                 |
| `Orders(id)/Close`        | `Orders_id_Close.yaml`           | `/Orders(id)/Close`             |
| `OrdersService_Preview`   | `OrdersService_Preview.yaml`     | `/OrdersService_Preview`        |

**Rule:** `(id)` → `_id`, `/` → `_`.  Single-entity paths (format `Entity(id)`
with no action) use `{id}` as the OpenAPI parameter placeholder; action paths
keep the literal `(id)` because a separate id `parameter` block is not required.

### Request body schema

The JSON body from the `<pre>` example is parsed and recursively converted to an
OpenAPI `schema` object.  Objects become `{type: object, properties: {…}}`;
arrays become `{type: array, items: {…}}`; scalars become
`{type: string, example: "…"}`.
